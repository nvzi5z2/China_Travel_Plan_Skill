# 反幻觉与数据完整性规则

> 每次生成攻略前必读。违反任一条 = 产出不可用。

---

## 数据完整性规则（7 条）

### 规则 1：每条事实必须有来源
餐厅评分、人均价格、航班信息、门票价格、活动时间——每一条可验证的事实都必须能追溯到某次搜索结果。

### 规则 2：关键数据交叉验证
- **天气**：至少 2 个来源（中央气象台 + AccuWeather）
- **航班**：FlightAware/Airportia + 航司官网
- **活动/节庆**：至少 2 家本地媒体或官方公告

### 规则 3：冲突就标注，不要默默选一个
两个来源对同一家餐厅的评分不一样 → 写出区间，标注冲突。
```
✅ "大众点评 4.3 / 携程 4.7（评分存在差异）"
❌ "评分 4.5"（自行取平均 = 编造）
```

### 规则 4：没搜到就说没搜到
- 餐厅评分搜不到 → 写「评分暂缺」
- 航班准点率搜不到 → 不要编百分比
- 活动时间不确定 → 写「建议出发前确认」

### 规则 5：标注数据时间
天气、活动、航班都是时效性数据。2024 年的评分不等于 2026 年的水平。必须标注数据来源的年份。

### 规则 6：餐厅评分以大众点评为准
大众点评是中国餐饮评分的权威来源。携程/猫途鹰评分仅作参考，使用时标注平台。只有携程数据时写「（携程评分）」。

### 规则 7：信息不足时的处理
1. 告诉用户：「{具体项目}在搜索结果中未找到可靠来源，建议出发前自行核实」
2. 不要用训练数据填补
3. 用户坚持要参考信息 → 提供但标注「⚠️ 未经验证」

## 反模式（14 条）

### 内容质量

| # | ❌ 不要做 | ✅ 应该做 |
|---|---------|---------|
| A1 | 在游客主街上推荐「秘制」「祖传」之类没有具体店名的模糊推荐 | 给出具体店名、方位、评分 |
| A2 | 2 人以上同行还建议打车而非高铁，当高铁明显更省时 | 对比时间和费用，给最优方案 |
| A3 | 女伴同行但行李清单没有女性用品 | 按人头分包：共享 / 男士 / 女士 |
| A4 | 只说「去 XX 景点」不解释为什么值得去 | 每条推荐附一句「为什么」 |

### 防幻觉

| # | ❌ 不要做 | ✅ 应该做 |
|---|---------|---------|
| B1 | 没搜到评分/价格时编一个 | 写「暂缺」 |
| B2 | 两个来源数据冲突时默默选一个 | 写出区间，标注冲突 |
| B3 | 用训练数据里的「常识」替代时效信息 | 搜不到就说搜不到 |
| B4 | 声称有来源但实际搜索结果里没有 | 每条引用都来自真实的搜索结果 |
| B5 | 把 2024 年的数据当最新呈现 | 标注数据年份 |

### HTML/UX

| # | ❌ 不要做 | ✅ 应该做 |
|---|---------|---------|
| C1 | 生成需要滚动整个页面才能看完的 HTML | 用底部 Tab 分栏 |
| C2 | 在日期标题上写 `onclick` 行内事件 | 用声明式 JS 通过 `addEventListener` 绑定 |
| C3 | 引入外部 CSS/JS 框架 | 纯 CSS + 原生 JS，零外部依赖 |
| C4 | 信息堆在一页不区分层级 | 可折叠卡片 + 标签页 |
| C5 | 餐厅卡片没有地图链接 | 餐厅名链接到高德搜索 |

### flyai 数据使用规则（新增）

| # | ❌ 不要做 | ✅ 应该做 |
|---|---------|---------|
| D1 | 把 flyai 返回的第一个结果直接当推荐 | LLM 逐个评估后选最适合画像的，附带推理 |
| D2 | 因为 flyai 没返回大床房就告诉用户「没有大床房」 | 保留所有房型选项，让 LLM 判断哪些可以变通（双床合并等） |
| D3 | 硬过滤——脚本直接排除不符合预设条件的选项 | 宽搜索→全量结构化→LLM 推理筛选。不让脚本替用户做决定 |
| D4 | 只推荐一家酒店/一个航班 | 给出 Top 2-3 + 各自的推荐理由和权衡，让用户选 |
| D5 | 攻略中不标注数据查询时间 | 标注「🕐 价格查询于 {ISO timestamp}，出发前建议重新确认」 |
| D6 | 用 flyai 数据编造评分或设施 | flyai 数据也是数据源，同样遵守规则 1-7 |
| D7 | 忽略 flyai 返回的 product_url | 每个推荐项的可预订按钮必须链接到真实的 product_url |
